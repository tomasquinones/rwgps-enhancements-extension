# Repository Guidelines

## Project Structure & Module Organization

This is a dependency-free Manifest V3 browser extension for `ridewithgps.com`. `manifest.json` defines extension metadata, permissions, and content script load order. `content/` contains injected JavaScript and CSS; `content/content.js` must load first because it installs the cross-browser `browser` shim, `content/shared.js` holds shared helpers, and feature modules live in files such as `speedcolors.js`, `climbs.js`, `weather.js`, and `calendar.js`. `popup/` contains the toolbar popup UI. `icons/` stores extension icon assets, `store/` stores listing/privacy copy, and `dist/` is generated by the build script.

## Build, Test, and Development Commands

- `./build.sh`: rebuilds `dist/`, stages extension files, and creates Firefox, Chrome, and source ZIP packages.
- Firefox local load: open `about:debugging#/runtime/this-firefox`, choose "Load Temporary Add-on", and select `manifest.json`.
- Chromium local load: open `chrome://extensions`, enable Developer mode, choose "Load unpacked", and select the repo root.

After source changes, reload the extension from the browser extension page. There is no `npm install`, bundler, or package manager workflow.

## Coding Style & Naming Conventions

Use plain JavaScript, 2-space indentation, semicolons, and the style already present in the edited file. Most modules use an IIFE with `"use strict"`. Use `browser.*` APIs, not direct `chrome.*` calls; Chromium compatibility depends on the shim. Prefix injected CSS classes with `rwgps-`. Use camelCase for variables and storage keys, with feature toggles named like `weatherEnabled`. Keep feature files lowercase and descriptive, for example `publiclands.js`.

## Testing Guidelines

No automated test suite is currently configured. Validate changes manually in Firefox and one Chromium browser. Exercise the relevant RWGPS page types, such as `/dashboard`, `/calendar`, `/trips/:id`, `/routes/:id`, and the planner. For UI features, test popup toggles, SPA navigation, cleanup when leaving a page, and persistence through `browser.storage.local`.

## Commit and Pull Request Guidelines

Recent commits use concise imperative subjects such as `Add Layers menu: Public Lands, Weather Radar, Wildfires`, `Fix Enhancements button placement on routes/trips`, and `Refactor: ...`. Release-related commits often include `release vYYYYMMDDa` or `Bump manifest to YYYYMMDDa`. Pull requests should include a short behavior summary, affected pages/features, manual browser test notes, screenshots for visual changes, and explicit notes for any `manifest.json` permission or version changes.

## Security and Configuration Tips

Do not add secrets or API keys. Keep `host_permissions` as narrow as practical. MV3 forbids inline scripts in extension HTML, so add external script files instead.
